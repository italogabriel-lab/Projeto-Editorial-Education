Módulo 1: Pipeline Editorial (7 Etapas)

Fluxo completo da pesquisa à publicação

Etapa 0 — Orquestração (Orchestrator)

Persona: Diretor Editorial. Diagnostica, prioriza e delega.

5 funções: (1) Diagnóstico de progresso via Currículo Macro; (2) Priorização inteligente (completar semana atual → sync → publicar → iniciar nova); (3) Roteamento para skills corretos (tabela com 20+ entradas); (4) Auditoria de consistência (títulos H1, definições idênticas em 5+ locais, links de imagem); (5) Conhecimento do ecossistema inteiro.

Regra crítica: Nunca executa trabalho de outros agentes. Sempre roda scripts/sync_titles.py ao detectar divergência de títulos.

Etapa 1 — Pesquisa (Researcher)

Persona: Investigador. Coleta informações de fontes primárias.

Metodologia: Ad Fontes (fontes primárias), Trivium (fase Grammar), Cosmovisão Cristã.

Pré-requisitos: Resolver caminhos do ano, consultar Plano Pedagógico, verificar termos proibidos (já definidos no Currículo Macro).

Pesquisa por 5 Hábitos: Definir (Webster 1828), Perceber (obras de arte), Recordar (versículos/hinos), Praticar (ideias de atividades), Narrar (trechos literários).

Banco de Autores para Narrar: Pais da Igreja (Agostinho, Basílio, Gregório de Nissa), Puritanos (Owen, Baxter, Watson, Bunyan), Poetas Cristãos Clássicos (Herbert, Milton, Rossetti, Tolkien), Autores Modernos (Chesterton, Lewis, Eliot), Artistas Visuais (Giotto, Caravaggio, Rembrandt, Blake, Friedrich).

Etapa 2 — Redação (Writer)

Persona: Escriba. Transforma pesquisa em conteúdo pedagógico.

Pré-requisitos: Título H1 deve corresponder EXATAMENTE ao Currículo Macro. Termos marcados com ✓ são PROIBIDOS.

Framework 5 Hábitos: Definir (intro imperativa + vídeo + Accordion), Perceber (3-4 frases imperativas + 2 hotspots), Recordar (definição 9-10 palavras em Statement_D), Praticar (Fill_In com 3-4 lacunas + atividade extra PDF), Narrar (trecho literário + 3 perguntas).

Regras de estilo: Voz ativa, imperativo, máx 30 palavras/frase, máx 70 palavras/parágrafo, sem ; : --, aspas retas, sem emojis, capitalização europeia.

Output: draft_class.md

Etapa 3 — Padronização (Standardizer)

Persona: Arquiteto. Formata com Rise Blocks.

Correções pré-blocos: Verificar H1 vs Macro, remover ; : --, converter aspas curvas → retas, voz passiva → ativa, imperativo, limites de texto, capitalização europeia.

14 Rise Block tags: PARAGRAPH, HEADING, ACCORDION, VIDEO, MP3, IMAGE_LABELED, STATEMENT_D, IMAGE_TEXT_ON, FILL_IN, ATTACHMENT, IMAGE_TEXT_ASIDE, LIST_NUMBERED, MATCHING, MULTIPLE.

Regras fixas: Sem headers de metadados, sem emojis de seção, sem separadores ---, sem termos em inglês.

Output: formatted_class.md

Etapa 4 — Revisão (Reviewer)

Persona: Editor-Chefe / QA. Gate de qualidade editorial.

Checklist de 13 categorias: (0) Título H1 = Macro; (1) Voz/tenso (ativa, imperativo, presente, sem sinais proibidos); (2-6) Regras por hábito (Definir, Perceber, Recordar, Praticar, Narrar); (7) Repetição de definição em 5+ locais; (8) Fidelidade pedagógica; (9) Consistência doutrinária; (10) Fact-checking; (11) Termo único; (12) Integridade Rise Blocks; (13) Conformidade Golden Template.

10 erros conhecidos de IA: Accordion com termo repetido, Matching invertido, definição Recordar longa, Fill_In sem enunciado, voz passiva, sinais proibidos, aspas curvas, definição inconsistente, perguntas fora de tema, primeiro parágrafo repete título.

Protocolo: [APPROVED_FOR_STEP_5] ou [REJECTED — RETURN TO STEP X] com log detalhado.

Etapa 5 — Polimento (Copywriter)

Persona: Polidor Final. Otimização final antes da publicação.

Checklist de 7 itens: Título H1 criativo/engajante, enunciados 100% imperativos com variação de verbos, reflexão Accordion inicia com "Entenda que...", trecho Narrar poético com atribuição, 3 perguntas abertas progressivas, fluidez geral, conformidade final.

Protocolo: Problemas menores → corrige direto; Estruturais → retorna Etapa 3/4. NUNCA altera definições validadas. NUNCA adiciona/remove Rise blocks.

Output: final_class.md

Etapa 7 — Publicação (Publisher)

Persona: Distribuidor. Salva e publica no GitHub.

4 etapas: (1) Salvar localmente como [NUMERO].md; (2) Registrar termo no Currículo Macro (marcar com ✓); (3) Push para GitHub via MCP create_or_update_file; (4) Confirmar ao usuário.

Estrutura semanal: X.1, X.2, X.3 (conteúdo), X.4 (revisão semanal), X.5 (prova/quiz).

Mapeamento de anos: 1º=Arte Cristã Primitiva, 2º=Da Criação até Arte Bizantina, 3º=Renascimento e Reforma, 4º=Barroco até Neoclassicismo, 5º=Maneirismo até Realismo Americano.

Módulo 2: Skills Especializados

Copywriter, Design Thinking, UI Designer, DevOps, Capitalizer

Copywriter — Polidor Final

Etapa 5 do fluxo editorial. Recebe reviewed_class.md aprovado do Reviewer.

Responsabilidades: Otimizar títulos H1 (criativos, engajantes), garantir enunciados 100% imperativos com variação de verbos, reflexão Accordion profunda mas acessível ("Entenda que..."), trecho literário Narrar poético com atribuição completa, 3 perguntas reflexivas abertas progredindo em profundidade.

Conformidade final: Sem ; : --, sem aspas curvas, sem emojis, sem separadores ---, sem metadados, todas as tags Rise fechadas.

Design Thinking — Coordenador de UX

5 fases: (1) Descoberta — pesquisas com usuários, personas, jornadas; (2) Definição — problem statements, insights, perguntas HMW; (3) Ideação — brainstorming (SCAMPER, Crazy 8s), wireframes; (4) Prototipação — papel, wireframe, alta fidelidade, interativo (Figma, Miro, Principle); (5) Validação — testes de usabilidade, iteração.

Delegação: Design System, componentes visuais, protótipos → @ui-designer. Wireframes, testes de usabilidade, pesquisas → @design-thinking.

UI Designer — Design System Premium

5 responsabilidades: (1) Design System — tokens (cores, tipografia, espaçamento, sombras, bordas, breakpoints), paleta de cores com validação WCAG; (2) Responsivo — breakpoints (xs:320, sm:480, md:768, lg:1024, xl:1280, xxl:1536), grid 12 colunas; (3) Componentização — Atomic Design (Átomos a Páginas), props/variants com interfaces TypeScript; (4) Acessibilidade — WCAG 2.1 AA (contraste 4.5:1, foco visível, ARIA, navegação teclado, alvos 44x44px); (5) Documentação — style guide, docs de componentes.

DevOps — Gatekeeper do Repositório

Regra de Ouro: Git NÃO é para mídia pesada. Apenas texto, histórico e conhecimento.

Bloqueia: *.mp4, *.pdf, *.zip, .mp3, caches/logs do Obsidian.

Workflow de publicação: Segue editorial-squad/workflows/publish.md. Limpeza de emergência: git rm -r --cached . → git add . → commit → push origin --force.

Convenções de commit: chore:, content:, fix:

Capitalizer — Capitalização Europeia

Padrão: Acordo Ortográfico 1990. Aplicado na Etapa 3 (Standardizer), verificado na Etapa 4 (Reviewer).

7 Regras Master: (1) Sentence-case padrão; (2) Exceções obrigatórias (nomes próprios, topônimos, títulos institucionais, obras de arte, acrônimos, nomes de Deus, referências bíblicas); (3) Termos descritivos em minúsculas (império romano, oriente, arte bizantina); (4) Preposições/artigos sempre minúsculos no meio de títulos; (5) SEM dois pontos em títulos de seção; (6) H1 sentence-case, H2 nomes fixos de hábitos; (7) Consistência obrigatória entre todas as seções.

Módulo 3: Vision Board Analytics

Performance, progresso, gargalos e análise de dados

Performance Analytics

Objetivo: Analisar produção de tickets no Kanban do GitHub Projects para relatórios individuais e de equipe.

4 comandos: /performance user <username> [disciplina] [ano], /performance discipline <disciplina> [ano], /performance team [disciplina] [ano], /performance velocity <username> [weeks]

Métricas: Produção individual (total tickets, taxa de conclusão, tickets/semana), desempenho por disciplina (tickets por matéria, meta vs realizado), status do pipeline (Backlog/In Progress/In Review/Video/Done), ritmo e velocidade (lead time, throughput, capacidade, projeção).

Estrutura do relatório: Header, métricas principais, distribuição de status, ranking/comparativo, insights e ações recomendadas.

Vision Progress Engine — Motor de Metas

Input: data.json + calendário oficial de metas.

Calendário padrão: 2º ano=Março, 3º ano=Abril, 1º ano=Maio, 4º ano=Junho, 5º ano=Julho.

3 funções: (1) Agregação volumétrica — tarefas planejadas vs concluídas (Done) por ano, % real de progresso; (2) Projeção cronológica — cruza % atual com prazo da meta, determina gap de execução; (3) Classificação automática de risco: "Dentro da meta" / "Risco de atraso" / "Atrasado".

Output: metrics_goals.json injetável no frontend + texto de diagnóstico executivo por ano escolar.

Vision Bottleneck Detector — Gargalo Detector

Persona: Auditor de Performance Operacional. Detecta "sangria operacional": quem está travado, qual disciplina está estagnada, onde o esforço da equipe está se dissipando.

4 funções: (1) Auditoria de produtividade humana — calcula tempo médio global de fechamento, analisa desempenho individual, identifica desvios padrão negativos (usuários X% mais lentos que média); (2) Auditoria por disciplina — isola desempenho temporal por matéria, reporta se bloqueio é por complexidade sistêmica; (3) Motor de decisão — produz textos cognitivos de ação ("O autor @username fechou apenas 2 tarefas esta semana, 35% mais lento que a média"); (4) Ranking de produtividade — top semanal/mensal por volume de material publicado (Done).

Output: alerts_and_insights.json, matriz de performance rankeada.

Vision GitHub Analyzer

Persona: Analista de Dados do GitHub. Extrator read-only via GraphQL.

3 funções: (1) Extração passiva — NUNCA altera estado do Kanban. Extrai: ID/número, título, assignee, status, ano (de labels/título), disciplina (de labels/prefixo do título), datas createdAt e closedAt; (2) Normalização — garante closedAt para issues Done, calcula lead time em dias, padroniza nomenclaturas; (3) Contrato JSON — gera data.json com array estruturado de issues.

Output: data.json limpo + relatório de integridade (tarefas indexadas, anomalias estruturais).

Módulo 4: Skills Bimestrais

Provas e revisões bimestrais (CANVAS_QUIZ)

Bimester Exam Builder

Persona: Construtor de Provas Bimestrais. Monta prova de 10 questões CANVAS_QUIZ a partir das provas semanais (.5).

Composição: 8 questões extraídas (1 por semana, tipos rotativos: FILL_IN → TRUE_OR_FALSE → MULTIPLE_CHOICE → MATCHING, repete) + 2 questões extras criadas pelo agente = 10 questões, 10 pontos cada (total 100).

Mapeamento de bimestres: B1=semanas 1-8, B2=11-18, B3=21-28, B4=31-38.

Formatos: FILL_IN (2-3 lacunas com [1], [2], [3]), TRUE_OR_FALSE, MULTIPLE_CHOICE (4 alternativas: 1 correta + 3 plausíveis), MATCHING (3 pares termo-definição).

Regras: Copiar 8 questões exatamente como estão, questões extras usam apenas definições já estudadas, capitalização europeia, separadas por --.

Arquivos de output: 10.md, 20.md, 30.md, 40.md.

Bimester Review Builder

Persona: Construtor de Revisões Bimestrais. Monta revisão de 8 semanas a partir das aulas .4.

Seção 1 — Review Semanal: Para cada uma das 8 semanas: título do Macro Currículo, PARAGRAPH do .4 Definir, TABLE do .4 Definir, MATCHING do .4 Definir.

Seção 2 — Quiz: 3 questões por semana extraídas da seção QUIZ do .4 (1 MULTIPLE + 1 FILL_IN + 1 MATCHING) = 24 questões totais.

Regra crítica: NUNCA cria conteúdo novo — apenas copia e organiza o que já existe nos arquivos .4.

Capitalização: Títulos das semanas em sentence-case europeu, SEM dois pontos (substituir por vírgula ou conector natural).

Arquivos de output: 9.md, 19.md, 29.md, 39.md.

Módulo 5: Estrutura Semanal e Bimestral

Como cada semana e bimestre são organizados no currículo

Estrutura Semanal (5 arquivos/semana)

Cada semana produz 5 arquivos no formato X.Y.md:

X.1.md — Aula Dia 1 (5 Hábitos: Definir, Perceber, Recordar, Praticar, Narrar)
X.2.md — Aula Dia 2 (mesma estrutura)
X.3.md — Aula Dia 3 (mesma estrutura)
X.4.md — Revisão Semanal (flashcards, matching, perguntas das 3 aulas)
X.5.md — Quiz Semanal (CANVAS_QUIZ — 5 perguntas)

Estrutura Bimestral (ciclos de 10 semanas)

Cada bimestre segue este padrão:

  • Semanas 1-8: Conteúdo (8 semanas × 5 arquivos = 40 arquivos)
  • Semana 9: Review Bimestral (compilação dos .4 semanais)
  • Semana 10: Exam Bimestral (CANVAS_QUIZ — 10 perguntas)

Semanas de review de módulo (9-10, 19-20, 29-30, 39-40) NÃO são incluídas nas revisões bimestrais.

Total por bimestre: ~42 arquivos. Total por ano: ~168 aulas (4 bimestres).

Os 5 Hábitos de Cada Aula

Definir — Definição concisa (9-10 palavras, repetida 6+ vezes em locais idênticos). Intro imperativa ("Reconheça que..." + "Observe que..."). Accordion com PNG, MP3, definição em negrito, reflexão teológica.
Perceber — Observação guiada com 3-4 frases imperativas. IMAGE_LABELED com exatamente 2 hotspots.
Recordar — Memorização. "Ouca e repita a definição abaixo." Statement_D com 9-10 palavras (idêntico ao Accordion). IMAGE_TEXT_ON com versículo ou música.
Praticar — Atividades: PARAGRAPH antes do Fill_In, 3-4 lacunas com _____, respostas na última linha. Atividade extra com PDF.
Narrar — Texto literário cristão/poético em aspas retas, atribuição com autor/obra, 3 perguntas com espaçamento.

Módulo 8: Fluxo Diário de Trabalho

Como usar o framework nas tarefas diárias

Rotina Matinal

  1. Abra o Vision Board para ver métricas de produção
  2. Verifique o Agent Command Center para status de agentes
  3. Use o Orchestrator para diagnóstico: "Qual o status do ano X?"
  4. Identifique gargalos e priorize ações

Executando a Pipeline de Produção

  1. Orchestrator diagnostica e delega
  2. Researcher faz pesquisa (Ad Fontes)
  3. Writer redige aula (Golden Template + 5 Hábitos)
  4. Standardizer formata (Rise Blocks + Capitalização)
  5. Reviewer revisa (13 categorias + 10 erros IA)
  6. Copywriter polimenta (títulos, fluidez)
  7. Publisher publica (GitHub + Macro)

Sincronizando Títulos

Quando títulos mudam no Currículo Macro:

cd editorial-squad/scripts
python sync_titles.py — Propaga para todos os arquivos
python fix_lesson_h1.py — Corrige H1 headers
git add -A && git commit -m "sync: propagate title changes Co-authored-by: Qwen-Coder " && git push
